Biomni -> MAS Implementation Plan (Code/Flow Planning)

0) Scope and objective
- Goal: Keep reusable Biomni runtime assets, replace orchestration flow with MAS defined in `MAS.txt` and prompt contracts in `prompt/*.txt`.
- Reuse-first policy: preserve tool implementations, execution runtime, and utility loaders; add an MAS control layer on top.

1) What to reuse from Biomni (do not rewrite first)
- Execution/runtime primitives:
  - `Biomni/biomni/tool/support_tools.py` (run_python_repl)
  - `Biomni/biomni/utils.py` (run_with_timeout, read_module2api, download helper)
- Agent loop references (for extraction, not direct MAS final form):
  - `Biomni/biomni/agent/a1.py`
  - `Biomni/biomni/agent/react.py`
- Config and paths:
  - `Biomni/biomni/config.py` (`BIOMNI_PATH`, `BIOMNI_DATA_PATH`, timeout, model settings)
- Tool catalog/registry:
  - `Biomni/biomni/tool/tool_registry.py`
  - `Biomni/biomni/tool/tool_description/*.py`

2) MAS control layer to add (new code structure)
- `Biomni/biomni/agent/mas_state.py`
  - state contract: user_query, selected_agents, domain_scores, final_master_plan, current_step_index, execution_history, replan_history
- `Biomni/biomni/agent/router_mas.py`
  - calls Router prompt and validates JSON: selected_agents, route_reason, act_required, domain_scores
- `Biomni/biomni/agent/domain_agent_mas.py`
  - Plan R1/R2/R3 + Execution R1/R2 adapters
- `Biomni/biomni/agent/orchestrator_mas.py`
  - R2.1/R3.1 integration + step loop + status branching
- `Biomni/biomni/agent/synthesizer_mas.py`
  - final answer synthesis for act_required true/false routes
- `Biomni/biomni/agent/prompt_runtime.py`
  - loads `prompt/Router Prompt.txt`, `prompt/Domain Agent Prompt.txt`, `prompt/Orchestrator Prompt.txt`
  - fills placeholders and enforces JSON-only outputs where required

3) Resource operation strategy decision

3.1 Option A: Direct use of `MAS_Resourse` only
- How: directly import each domain `env_desc.py` + `descriptions/*.py` + `tools/*.py` at runtime.
- Pros: simple concept, no index rebuild step.
- Cons: heavier prompt payload, harder to keep stable IDs/snapshots, weaker reproducibility.

3.2 Option B: Indexed + lazy resolution (`MAS_Resourse_`)
- How: Plan R1 reads thin catalog `MAS_Resourse_/indexes/master_index.json`; Execution resolves selected IDs to full specs via `MAS_Resourse_/scripts/resolve_selected_resources.py` logic.
- Pros: aligns with MAS Plan/Execution split, stable IDs, snapshot-friendly, lower token overhead.
- Cons: requires index freshness management and path consistency.

3.3 Recommended default for this project
- Choose Option B as default (indexed + lazy resolution).
- Keep Option A as fallback debug mode only.

5) Data lake local download policy (critical)
- Problem: some selected data_lake items are not guaranteed to already exist locally.
- Policy:
  1. Plan stage uses only metadata (name/short_description).
  2. Before Execution step starts, run preflight for selected data files.
  3. Missing files trigger on-demand fetch to local data lake path.
- Local path standard:
  - use Biomni config root from `BIOMNI_DATA_PATH` or `BIOMNI_PATH`
  - target: `<root>/biomni_data/data_lake`
- Reuse helper:
  - `Biomni/biomni/utils.py` `check_and_download_s3_files(...)`
- Execution guard:
  - if required file still missing after fetch, Verifier returns FAILURE with immediate_action for retry/fallback.

6) End-to-end flow to implement (MAS-aligned)

6.1 Router stage
- Input: user_query
- Prompt: `prompt/Router Prompt.txt`
- Output JSON: selected_agents, route_reason, act_required, domain_scores
- Branch:
  - act_required=false -> synthesizer path
  - act_required=true -> plan stages

6.2 Plan stages
- R1 (resource selection): domain agent selects tools/data/libraries IDs from thin index
- R2 (domain plan drafting): checklist + checklist_steps
- R2.1 (orchestrator draft integration): build draft_master_plan with single-owner policy
- R3 (domain critique): critique + recommended_changes
- R3.1 (orchestrator finalization): final_master_plan

6.3 Execution stages (per step)
- R1: assigned owner agent writes/runs code using resolved specs
- ACT: collect observe_output
- Handoff rule: previous_step_observe_output truncated to max 10000 chars
- R2 Verifier JSON:
  - status in {SUCCESS, FAILURE, PLAN_REVISION}
  - reason, immediate_action, observe_output
- Branching:
  - SUCCESS -> next step
  - FAILURE -> retry same step
  - PLAN_REVISION -> partial replan from target_step..end

7) Implementation phases and deliverables

Phase A (foundation)
- Deliverables:
  - mas_state model
  - prompt_runtime loader
  - router_mas with schema validation

Phase B (plan pipeline)
- Deliverables:
  - domain R1/R2/R3 adapters
  - orchestrator R2.1/R3.1 adapters
  - final_master_plan validator (step_id continuity, owner_agent required, success_criteria required)

Phase C (execution loop)
- Deliverables:
  - step runner with resolved specs
  - verifier adapter + branch control
  - context handoff truncation rule (10000)

Phase D (resource robustness)
- Deliverables:
  - data_lake preflight + on-demand download
  - missing file error policy
  - index snapshot per run for reproducibility

Phase E (evaluation)
- Deliverables:
  - baseline vs MAS run protocol
  - success/recovery/time/token metrics report

8) Operational decisions to lock before coding
- Decision 1: canonical index root path (`MAS_Resourse_/indexes`) and env var names
- Decision 2: index refresh trigger (manual command vs auto-on-start)
- Decision 3: data download source policy (S3 only vs mixed mirrors)
- Decision 4: retry/replan limits for safety

9) Recommended command-level workflow (developer runbook)
- Step 1: build or refresh index from source resources
- Step 2: run Router + Plan dry-run (no execution)
- Step 3: resolve selected IDs to full specs
- Step 4: preflight local data files and fetch missing assets
- Step 5: execute step loop with verifier branching
- Step 6: generate final answer + save run snapshot

10) Output rendering policy (internal JSON + human-readable messages)
- Core principle:
  - internal contracts stay strict JSON/dict by stage
  - user-facing output is rendered as readable stage/domain messages
- Render labels (recommended):
  - `[Router Message]`
  - `[Plan-R1 | <Domain> Message]`
  - `[Plan-R2 | <Domain> Message]`
  - `[Orchestrator R2.1 Message]`
  - `[Plan-R3 | <Domain> Message]`
  - `[Orchestrator R3.1 Message]`
  - `[Execution-R1 | <Domain> | Step <id> Message]`
  - `[Execution-R2 Verifier | <Domain> | Step <id> Message]`
  - `[Synthesizer Message]`
- Message payload policy:
  - keep schema fields in trace store (machine-readable)
  - show concise natural-language summaries to user
  - include key status values (SUCCESS/FAILURE/PLAN_REVISION) in verifier messages
- API behavior recommendation:
  - `agent.go(query)` returns final answer by default
  - `agent.go(query, verbose=True)` returns stage trace + final answer
  - `agent.go(query, stream=True)` emits stage/domain messages incrementally
- Safety and readability:
  - do not expose raw long logs directly by default
  - for observations, display short excerpt and keep full trace in artifacts
  - preserve previous_step_observe_output truncation rule (max 10000) for handoff

11) Summary decision
- Use `MAS_Resourse` as source-of-truth resource definitions.
- Use `MAS_Resourse_` as runtime index/resolve layer.
- Reuse Biomni execution engine and utilities.
- Replace orchestration with MAS state machine and round-specific prompts.

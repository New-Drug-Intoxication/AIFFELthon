3 Round - Critique
---------------------------------
  현재 당신은 {domain} 전문가 관점에서, 아래 사용자 질의를 해결하기 위한 통합 초안 계획을 검토합니다.

  [사용자 질의]
  {user_query}

  [검토 대상 초안 계획]
  {draft_master_plan}

  [중점 검토 관점]
   - [사용자 질의]에 연관된 것
   - 계획이 생물학적으로 타당한지
   - 방법론 간 순서와 연결이 일관적인지
   - 실제 실행 단계에서 실패 없이 수행 가능한지

  [지침]
   - 위 관점을 기준으로 문제점을 식별하고, 바로 반영 가능한 수정 문장으로 제안하십시오.

  [비평 강도 제한 규칙]
   - 당신의 역할은 초안 계획의 전면 재작성이 아니라, 실행 실패 가능성을 줄이기 위한 핵심 결함 보정입니다.
   - 기존 단계 구조와 질문 범위를 반드시 유지하십시오.
   - 최소한으로 제시하십시오.

  [수정 제안 작성 규칙]
   - recommended_changes는 즉시 반영 가능한 수정 지시 문장으로 작성하십시오.
   - 하나의 제안에는 하나의 변경 의도만 담으십시오.
   - 각 제안은 "압축된 수정 문장"으로 작성하고, 구현 디테일을 나열하지 마십시오.
   - 권장 제안 수는 1~3개입니다. (사소한 표현 개선은 제안하지 마십시오)

  [변경 우선순위]
   - 우선: 누락 입력 보완, 실행 순서 충돌 해소, 성공기준 명확화
   - 지양: 신규 데이터원·신규 파이프라인 대거 도입, 사실상 재기획에 해당하는 변경, 파라미터/임계값 명시
   - 금지: 기존 초안을 새 계획으로 대체하는 제안
   - 금지: 루프/조건문/예외처리/파일 저장/컬럼 매핑/상세 API 재시도 절차를 recommended_changes에 작성

  [범위 고정 규칙]
   - 비평은 [사용자 질의]의 직접 요구 범위 안에서만 수행
   - 질의에서 요구하지 않은 확장 산출물(장기 로드맵, 인프라 재설계, 광범위 후속연구)을 요구하지 마십시오.

  [MSA 출력 스키마]
   JSON만 출력:
   {
     "critique": "...",
     "recommended_changes": ["...", "..."]
   }

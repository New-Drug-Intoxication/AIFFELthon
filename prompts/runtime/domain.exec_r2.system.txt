  [Decision Guidelines (Choose One)]

  Based on the observation result, you must decide the next direction of the analysis.


   1. Success (SUCCESS):
       - `success_criteria` must be satisfied. If not satisfied, do not choose SUCCESS.
       - Code execution has no errors, and the target outputs (numeric values, tables, statistics, etc.) were produced correctly.
       - The outputs are biologically plausible, and enough information is secured to proceed to the next step.
       - Route: report so the orchestrator can proceed to the next step.


   2. Failure (FAILURE - code/output error):
       - Choose FAILURE if unmet `success_criteria` is caused by code/execution/environment issues.
       - Python/R/Bash syntax errors or library import errors occurred.
       - There is no error, but required outputs were not printed via `print()`, so results cannot be verified.
       - The logic is valid, but technical issues (e.g., path typos) can be fixed with simple edits.
       - Route: report that code should be corrected and retried (return to Execution Round 1).


   3. Step change (PLAN_REVISION - step-level strategy revision):
- Choose this only when unmet `success_criteria` is caused by problems in the plan/method/resource selection itself, not by execution errors.
- The assigned tool does not fit the current data format, or the analysis methodology itself is incorrect.
- A required file is missing in the data lake, or the current plan cannot solve the user's query.
- If the step/success_criteria itself requires additional user response/clarification (multi-turn interaction), choose PLAN_REVISION.
- In this system, "collect more inputs from user" is a planning defect, not a retriable execution failure.
- Classify the following as FAILURE (retry), not PLAN_REVISION:
  - schema/column mismatch, identifier mapping failure, inadequate normalization rules, inadequate missing-value handling
  - missing output format or missing required `print`
  - data preprocessing/parsing issues solvable by code changes within the same step
- Choose PLAN_REVISION only when it is clear that retrying the same step cannot resolve the issue.
- Route: request that the orchestrator revise the current step strategy. The orchestrator will then re-evaluate and update subsequent residual steps accordingly before resuming execution.

  [Strict Concision Contract]
   - For FAILURE or PLAN_REVISION, return only the core signal needed for the next attempt.
   - `reason` MUST be exactly one sentence and <= 140 characters.
   - `immediate_action` MUST be exactly one sentence and <= 160 characters.
   - Use this prefix format in `reason`: "<TYPE>: <core cause>".
   - Allowed `<TYPE>` values: RUNTIME_ERROR, SCHEMA_MISMATCH, DATA_MISSING, TOOL_UNAVAILABLE, METHOD_MISMATCH, USER_INPUT_REQUIRED.
   - `immediate_action` must describe only concrete next edits/actions for the next execution attempt (1-2 actions max).
   - Do not include long background, repeated context, caveats, or process narration.
   - If uncertain, choose the most likely core cause and keep output short; never expand into long explanation.
